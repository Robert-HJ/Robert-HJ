# 안녕하세요. 백엔드 개발자 홍혁준입니다.

Java/Spring Boot 기반 6년차 백엔드 개발자입니다.

운영 중 반복되는 문제를 공통 모듈과 구조로 정리하는 일에 관심이 많습니다. 특히 메시징 안정성, 실패 복구, 외부 연동 표준화, 데이터 정합성, 모듈을 사용하는 개발자가 이해하기 쉬운 인터페이스 설계를 중요하게 봅니다.

최근에는 위메이드에서 글로벌 플랫폼 WEMIX PLAY 백엔드를 개발·운영하며, 여러 서비스가 함께 사용하는 공통 모듈과 운영 개선 작업을 맡았습니다.

## 관심 있는 문제

- 반복되는 운영 작업을 재사용 가능한 백엔드 모듈로 만들기
- 메시지 유실, 중복 처리, 실패 복구 흐름을 추적 가능하게 만들기
- 외부 연동 지점을 인터페이스, 로그, 문서로 명확하게 정리하기
- 실제 측정 결과와 운영 조건을 기준으로 기술 선택하기
- 서비스 운영 경험을 바탕으로 플랫폼 백엔드와 AI 백엔드 영역으로 확장하기

## 주요 경험

### Kafka Pub/Sub 공통 모듈

Outbox 패턴을 기반으로 메시지 발행 이력, 재처리 흐름, 실패 알림, 수동 복구 API를 포함한 공통 메시징 모듈을 설계했습니다.

서비스 팀이 비즈니스 로직에 집중할 수 있도록, 메시지 발행 이력과 실패 복구 같은 공통 운영 로직을 모듈 내부로 감추는 것을 목표로 했습니다.

### 외부 연동 표준화

외부 서버와 통신하는 연동 구조를 표준화했습니다.

요청·응답 구조, 로깅, 예외 처리, 재시도 흐름을 비즈니스 코드와 분리해 신규 연동 초기 작업을 줄이고, 장애가 발생했을 때 어느 지점에서 문제가 생겼는지 추적하기 쉽게 만들었습니다.

### 금칙어 필터링 성능 개선

여러 문자열 탐색 방식을 직접 비교하고 벤치마크한 뒤, 다중 패턴 탐색에 적합한 Aho-Corasick 알고리즘을 적용했습니다.

단순히 검색 속도를 높이는 것뿐 아니라, 금칙어 갱신 방식, 장애 시 동작, 사용하는 팀이 알아야 할 운영 제약까지 함께 정리하는 데 중점을 두었습니다.

## 일하는 방식

반복되는 불편함을 발견하면, 그 문제를 한 번의 기능 개발로 끝내기보다 팀이 계속 사용할 수 있는 구조로 바꾸려는 편입니다.

또한 담당자가 명확하지 않은 일이라도 필요하다고 판단되면 먼저 정리하고 해결하려고 합니다.

- 어떤 부분을 모듈 내부로 감춰야 하는가?
- 어떤 부분을 인터페이스로 열어둬야 하는가?
- 실패했을 때 어떤 흔적이 남는가?
